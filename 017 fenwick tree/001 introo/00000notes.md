# Fenwick Tree (BIT): The Smart Prefix Sum

### 1. The Core Purpose
- Efficiently calculate **Prefix Sums** while allowing **Frequent Updates**.
- Both operations take **O(log N)**.

### 2. The "Last Set Bit" Magic
The "Physics" of the Fenwick Tree relies on isolating the rightmost `1` in the binary form: `x & -x`.
- **To Query Sum:** `index -= (index & -index)` (Move to parent). Parent is index less than current index so minus !!

- **To Update:** `index += (index & -index)` (Move to next responsible node)

### 3. Why it's better than Segment Trees?
- **Memory:** Uses exactly the same space as the original array ($O(N)$).
- **Code:** Much shorter and faster to write than a Segment Tree.
- **Constant Factor:** It is incredibly fast in practice due to simple bitwise operations.

### 4. The Limitation
- Only works for "Invertible" operations (like addition, multiplication, XOR).
- It is harder to find a "Range Minimum" with a Fenwick Tree compared to a Segment Tree.

# The Physics of `x & -x`

### 1. Two's Complement Logic
- `-x` is defined as `(~x + 1)`.
- Adding `1` to the inverted bits causes a "carry" that stops exactly at the first `1` from the right.
- This makes the rightmost `1` the only bit that is identical in both `x` and `-x`.

### 2. The Result
- `x & -x` isolates the **lowest power of 2** that divides `x`.
- **Even numbers:** Returns the largest power of 2 that is a factor (e.g., for 12, it's 4).
- **Odd numbers:** Always returns 1.

### 3. The Fenwick Application
- **Subtraction (`-=`)** strips the bit to move to the previous "independent bag" of sums.
- **Addition (`+=`)** adds the bit to "expand" your index until it hits the next larger range of responsibility.
# The Secret Sauce: `idx & -idx`

This bitwise trick is what finds the **Last Set Bit** (the rightmost `1`). 

### Why it works:
- Suppose `idx = 6` (Binary `0110`).
- `-idx` (Two's Complement) is `1010`.
- `0110 & 1010` = `0010` (Decimal **2**).

### 1. In Update (`+=`):
- `6 + 2 = 8`.
- We move from the manager (6) to the CEO (8).

### 2. In Query (`-=`):
- `6 - 2 = 4`.
- We jump from the sum of elements [5,6] to the sum of elements [1,2,3,4].

# What is an Invertible Operation?

### 1. Definition
An operation is invertible if there exists an inverse operation that can "cancel out" its effect.

### 2. The Logic
- **Sum:** Inverse is Subtraction. `(A + B) - B = A`.
- **XOR:** Inverse is XOR. `(A ^ B) ^ B = A`.
- **Max/Min:** No inverse. If you remove the Max, you don't know what the *new* Max is without re-scanning.

### 3. The Impact on DSA
- **Fenwick Trees** REQUIRE invertible operations because they rely on `Prefix(R) - Prefix(L-1)`.
- **Segment Trees** do NOT require invertibility. They store the answer for the left child and right child separately, so they can just combine them: `min(left_child, right_child)`.

# When to pull out the Fenwick Tree

- [x] **Scenario:** You need Prefix Sums + Point Updates.
- [x] **Scenario:** You need to count **Inversions** (how many numbers to the right are smaller).
- [x] **Scenario:** You are in a contest and only have 2 minutes left (It's faster to type than a Segment Tree).
- [ ] **Avoid if:** You need to find **Range Minimum/Maximum** (Segment Tree is much better for "Non-Invertible" operations).
# Fenwick Tree vs. Segment Tree

Both Fenwick Trees (Binary Indexed Trees) and Segment Trees are designed to solve "Range Query" problems, allowing you to update values and query ranges (sums, mins, maxes) in $O(\log N)$ time.

# Fenwick vs. Segment Tree: The Choice

### 1. The Rule of Thumb
- Can you "undo" the operation with subtraction/division? 
  - **Yes** (Sum, XOR) -> **Fenwick Tree** (Simpler/Faster).
  - **No** (Min, Max, GCD) -> **Segment Tree** (More Powerful).

### 2. The Lazy Factor
- Do you need to update a whole range at once? 
  - If yes, **Segment Tree with Lazy Propagation** is the only way to stay $O(\log N)$.

### 3. Complexity
- **Time:** Both are $O(\log N)$ per operation.
- **Space:** - Fenwick: $N$
  - Segment: $4N$

### Quick Comparison Table

| Feature | Fenwick Tree (BIT) | Segment Tree |
| :--- | :--- | :--- |
| **Best For** | Prefix sums and point updates | General range queries (min/max/GCD) |
| **Space Complexity** | $O(N)$ (minimal memory footprint) | $O(4N)$ (needs extra nodes) |
| **Update Time** | $O(\log N)$ (faster constant factors) | $O(\log N)$ |
| **Range Updates** | Complex to implement | Easy with **Lazy Propagation** |
| **Implementation** | Short, iterative, non-recursive | Longer, usually recursive |
| **Operations** | Only invertible (Sum, XOR, etc.) | Any associative (Min, Max, GCD, etc.) |

---

### 1. Fenwick Tree (Binary Indexed Tree)
The Fenwick Tree is a compact data structure that stores cumulative sums. It exploits the binary representation of an index to manage ranges.

* **How it works:** Each node at index `i` is responsible for a range of elements ending at `i`. The size of that range is determined by the "least significant bit" (LSB) of the index.
* **The "Invertible" Constraint:** To find the sum of range $[L, R]$, a BIT calculates `Sum(0 to R) - Sum(0 to L-1)`. This works for addition, but it **cannot** find the minimum or maximum in a range because you cannot "subtract" a minimum to isolate a sub-range.
* **Pros:** It is extremely memory-efficient and very fast in practice due to small constant factors. It is often the preferred choice for simple prefix sum problems.

### 2. Segment Tree
A Segment Tree is a full binary tree where each leaf represents an element, and each internal node represents the combined result of its children (a "segment").

* **How it works:** It recursively divides the array into halves until it reaches individual elements. Queries are answered by combining the pre-calculated results of these segments.
* **Versatility:** Unlike the BIT, it doesn't need to "subtract" ranges. To find the minimum in $[L, R]$, it simply identifies the specific segments that cover that range and takes the minimum of those values.
* **Lazy Propagation:** This is a major advantage. It allows you to update an entire range (e.g., "add 10 to every element from index 5 to 50") in $O(\log N)$ time. Fenwick Trees typically require $O(N \log N)$ or complex modifications for this.

---

### Which one should you use?

* **Use a Fenwick Tree if:** You only need **Range Sum Queries** and **Point Updates**, or if you are constrained by strict memory limits.
* **Use a Segment Tree if:** You need to handle **Range Minimum/Maximum Queries**, or if you need to perform efficient **Range Updates**.

# The Fenwick Tree: The "Responsibility" Formula

To know exactly which range a node is responsible for, you look at its **Binary Signature**. The "Physics" rule is this: 

> **A node at index $i$ is responsible for the range $[i - RSB + 1, i]$**
> *(Where $RSB$ is the value of the Rightmost Set Bit: `i & -i`)*

---

### 1. The "Subtraction" Rule
To find the starting point of a node's responsibility, you simply subtract its $RSB$ from its index and then add 1.

| Index ($i$) | Binary | $RSB$ (`i & -i`) | Range Calculation | **Range of Responsibility** |
| :--- | :--- | :--- | :--- | :--- |
| **1** | `0001` | **1** | $(1 - 1 + 1)$ to $1$ | **[1]** |
| **2** | `0010` | **2** | $(2 - 2 + 1)$ to $2$ | **[1, 2]** |
| **3** | `0011` | **1** | $(3 - 1 + 1)$ to $3$ | **[3]** |
| **4** | `0100` | **4** | $(4 - 4 + 1)$ to $4$ | **[1, 2, 3, 4]** |
| **6** | `0110` | **2** | $(6 - 2 + 1)$ to $6$ | **[5, 6]** |
| **8** | `1000` | **8** | $(8 - 8 + 1)$ to $8$ | **[1, 2, 3, 4, 5, 6, 7, 8]** |

---

### 2. The Visual Intuition: "Power of Zeros"
Think of the binary trailing zeros as the **Power** of the manager.

* **Index 7 (`0111`):** No trailing zeros. $RSB = 1$. It’s a **"Front-line worker."** It only knows about itself.
* **Index 6 (`0110`):** One trailing zero. $RSB = 2$. It’s a **"Team Lead."** It knows about 2 people (itself and the person before it).
* **Index 4 (`0100`):** Two trailing zeros. $RSB = 4$. It’s a **"Manager."** It knows about 4 people.
* **Index 8 (`1000`):** Three trailing zeros. $RSB = 8$. It’s a **"CEO."** It knows about 8 people.

---

### 3. Why this matters for the `query` loop?
When you do `sum += bit[i]` and then `i -= (i & -i)`, you are literally jumping over the range that the current node has already accounted for.

**Example: Sum(7)**
1.  **Look at Index 7 (`0111`):** Its range is `[7, 7]`.
2.  **Subtract $RSB$ ($7-1$):** Now you are at Index 6.
3.  **Look at Index 6 (`0110`):** Its range is `[5, 6]`.
4.  **Subtract $RSB$ ($6-2$):** Now you are at Index 4.
5.  **Look at Index 4 (`0100`):** Its range is `[1, 4]`.
6.  **Subtract $RSB$ ($4-4$):** You are at 0. **Stop.**

**Result:** By jumping to the "start" of each manager's range, you collected every number exactly once: **[7] + [5,6] + [1,2,3,4]**.

# The Fenwick Tree: The Physics of "Binary Responsibility"

A Fenwick Tree (Binary Indexed Tree) replaces the slow $O(N)$ Prefix Sum update with a system where each index has a specific, limited range of responsibility based on **Binary Math**.

---

### 1. The Core Idea: "Binary Responsibility"

Every index in a Fenwick Tree is responsible for a range of elements. The size of that range is determined by the **value of its rightmost set bit**.

* **Index 8 (Binary `1000`):** The rightmost set bit is $2^3 = 8$. It is responsible for **8 elements** (1 through 8). It's a **"Big Boss."**
* **Index 6 (Binary `0110`):** The rightmost set bit is $2^1 = 2$. It is responsible for **2 elements** (5 and 6). It's a **"Middle Manager."**
* **Index 7 (Binary `0111`):** The rightmost set bit is $2^0 = 1$. It is responsible for only **1 element** (itself). It's a **"Front-line Worker."**

---

### 2. The Two Movements (The Physics)

#### **The Query: "Collecting the Bags"**
To find the sum from 1 to 7, you don't look at 7 individual people. You collect the "bags" of pre-calculated responsibility:

1.  **Go to 7 (`0111`):** It holds its own value. **Collect it.**
2.  **Remove the last bit:** $7 - 1 = 6$.
3.  **Go to 6 (`0110`):** It holds the sum of 5 and 6. **Collect it.**
4.  **Remove the last bit:** $6 - 2 = 4$.
5.  **Go to 4 (`0100`):** It holds the sum of 1, 2, 3, and 4. **Collect it.**
6.  **Stop:** $4 - 4 = 0$.

**Total Sum = Bag(7) + Bag(6) + Bag(4).** You only touched **3 nodes** instead of 7!

#### **The Update: "Telling the Bosses"**
If you update index 3, you only inform the managers whose range **includes** index 3. The news "ripples up" the chain:

1.  **Update 3 (`0011`).**
2.  **Find the boss (Add last bit):** $3 + 1 = 4$.
3.  **Update 4 (`0100`).**
4.  **Find the boss (Add last bit):** $4 + 4 = 8$.
5.  **Update 8 (`1000`).**

This ripple effect ensures updates stay within **$O(\log N)$** time.

---

### 3. The "Magic" Formula: `x & -x`

How do we isolate that **"Rightmost Set Bit"**? We use bitwise logic. In binary, `-x` is the two's complement of `x`. When you `AND` them together, only the lowest `1` survives.

* **Example for 6:**
    * `6` is `0110`
    * `-6` is `1010`
    * `6 & -6` is `0010` (which is **2**).
* **Physics:**
    * To **Query**: `idx -= (idx & -idx)` (Jump to the next block).
    * To **Update**: `idx += (idx & -idx)` (Climb to the next boss).

```cpp

#include <vector>
#include <iostream>

using namespace std;

class FenwickTree {
    vector<int> bit;
    int n;

public:
    FenwickTree(int size) {
        n = size;
        bit.assign(n + 1, 0);
    }

    // Update: Telling the Bosses (Moving UP)
    // Uses a simple for loop with the x & -x jump
    void update(int i, int delta) {
        for (; i <= n; i += i & -i) {
            bit[i] += delta;
        }
    }

    // Query: Collecting the Bags (Moving DOWN)
    // Uses a simple for loop with the x & -x jump
    int query(int i) {
        int sum = 0;
        for (; i > 0; i -= i & -i) {
            sum += bit[i];
        }
        return sum;
    }

    // Range Sum: [L, R]
    int queryRange(int l, int r) {
        return query(r) - query(l - 1);
    }
};

int main() {
    vector<int> nums = {1, 3, 5};
    FenwickTree ft(nums.size());

    // Build
    for (int i = 0; i < nums.size(); i++) {
        ft.update(i + 1, nums[i]);
    }

    cout << "Sum [1, 3]: " << ft.queryRange(1, 3) << endl; // 1+3+5 = 9
    
    ft.update(2, 2); // Change 3 to 5 (delta is +2)
    cout << "New Sum [1, 3]: " << ft.queryRange(1, 3) << endl; // 1+5+5 = 11

    return 0;
}
```
# Spotting the Difference: Build Complexity

### 1. The O(N log N) Way (What you have above)
- **Code:** `for (i=0 to N) { update(i, val); }`
- **Physics:** Each brick is carried individually from the ground to its final height in the skyscraper.
- **Complexity:** $N$ elements $\times$ $\log N$ climb height.

### 2. The O(N) Way (The Ripple)
- **Code:** One loop through the array where `bit[parent] += bit[child]`.
- **Physics:** We build the first floor, then push the whole floor into the second floor, then push that into the third. No one has to climb more than one level at a time.
- **Complexity:** Each element is visited exactly once.

# O(n) build very clean

```cpp

#include <iostream>
#include <vector>

using namespace std;

// O(N) Build: The "Ripple Effect"
void buildBIT(vector<int>& bit, const vector<int>& nums, int n) {
    // 1. Initial Load: Put values into the tree (1-indexed)
    for (int i = 1; i <= n; i++) {
        bit[i] = nums[i - 1];
    }

    // 2. The Linear Push: Every node hands its bag to its immediate boss
    for (int i = 1; i <= n; i++) {
        int parent = i + (i & -i); // The "Secret Sauce" inline
        if (parent <= n) {
            bit[parent] += bit[i];
        }
    }
}

// O(log N) Update: Climbing Up to the Bosses
void update(vector<int>& bit, int n, int i, int delta) {
    for (; i <= n; i += (i & -i)) {
        bit[i] += delta;
    }
}

// O(log N) Query: Jumping Backwards to Collect Bags
int query(const vector<int>& bit, int i) {
    int sum = 0;
    for (; i > 0; i -= (i & -i)) {
        sum += bit[i];
    }
    return sum;
}

int main() {
    vector<int> nums = {1, 3, 5, 7, 9, 11};
    int n = nums.size();
    
    // Size is always n + 1 for 1-based indexing
    vector<int> bit(n + 1, 0);

    // Build the tree in O(N)
    buildBIT(bit, nums, n);

    cout << "Initial Sum (first 4): " << query(bit, 4) << endl; // 1+3+5+7 = 16

    // Update index 2: value changes from 3 to 10 (delta is +7)
    update(bit, n, 2, 7);
    
    cout << "Sum after update (first 4): " << query(bit, 4) << endl; // 16 + 7 = 23

    return 0;
}
```
# The Query Physics: Deconstructing the Prefix Sum

When you perform a **Query**, you are essentially "peeling the onion" from the outside in. You start at your target index and jump **backwards** to collect the pre-calculated "bags" of sums.

---

### 1. The Physics of the Query Jump
When you execute `i -= (i & -i)`, you are moving from a higher range to a lower range that is completely independent.

**Example: Sum up to 7**

* **Step 1: Index 7 (`0111`)**
    * **Range:** `[7, 7]`
    * This is a "Front-line worker" bag. It only knows about itself.
    * *The Jump:* `7 - (7 & -7)` $\to$ `7 - 1 = 6`.
* **Step 2: Index 6 (`0110`)**
    * **Range:** `[5, 6]`
    * This "Middle Manager" bag covers the next two elements.
    * *The Jump:* `6 - (6 & -6)` $\to$ `6 - 2 = 4`.
* **Step 3: Index 4 (`0100`)**
    * **Range:** `[1, 4]`
    * This "Big Boss" bag covers everything from 1 to 4.
    * *The Jump:* `4 - (4 & -4)` $\to$ `4 - 4 = 0`. **Stop.**

**Result:** By jumping to the "start" of each manager's range, you collected every number exactly once: **[7] + [5, 6] + [1, 2, 3, 4]**.

---

### 2. Why "Independent"?
This is the most critical part of the Fenwick Tree's efficiency. Notice the segments we collected:
1.  `[7, 7]`
2.  `[5, 6]`
3.  `[1, 4]`

**The Beauty of the Math:**
* **No Overlaps:** You never count the same index twice.
* **No Gaps:** You never miss a single number between 1 and 7.
* **Perfect Efficiency:** The binary math `i - (i & -i)` acts as a perfect **Range Jumper**, ensuring you land exactly at the start of the next pre-calculated block in $O(\log N)$ steps.

---

### 3. Summary for your Notes

| Action | Direction | Bitwise Op | Goal |
| :--- | :--- | :--- | :--- |
| **Update** | Upward (Higher Indices) | `i += (i & -i)` | Tell the bosses about a change. |
| **Query** | Downward (Lower Indices) | `i -= (i & -i)` | Collect independent sum blocks. |


# The Query Physics: Deconstruction

### 1. The Direction
- You start at the **End** (index `i`).
- You jump **Backwards** (minus).
- Each jump takes you to the **start of the previous manager's territory**.

### 2. The Binary Interpretation
- `7` is `4 + 2 + 1`.
- A Fenwick Query for 7 literally asks for:
  - The bag of size **1** (at index 7)
  - The bag of size **2** (at index 6)
  - The bag of size **4** (at index 4)
- You are simply summing up the **powers of 2** that make up your index!

### 3. Efficiency
- Since any number $N$ can be represented by at most $\log_2 N$ bits, you will never make more than $\log N$ jumps to find any sum.

`bit[parent] += bit[i]`
- This is correct. You are "donating" your accumulated sum to your immediate supervisor.

### 3. Why not the other way?
If you did `bit[i] += bit[parent]`, you would be trying to take values from the future (indices we haven't processed yet). That would break the linear flow.

Currently, your constructor calls update() in a loop. This is $O(N \log N)$. While this is totally fine for most interviews, if you want to impress a Google or Uber interviewer, you can mention that you can build the tree in $O(N)$.The Logic: Each index i is responsible for its own value. After adding its value, it just passes its entire "current total" to its immediate boss (i + rMsB(i)).

# Why the O(N) "Ripple" Works

### 1. The Single-Step Rule
In the $O(N)$ build, index $i$ only talks to its **immediate** parent $j = i + (i \& -i)$. 

### 2. The Chain Reaction
- Because the loop moves **forward** ($1 \to N$), every node is fully "filled" by its subordinates before it hands its own total to its boss.
- **Example:** - Node 1 hands to 2.
    - Node 2 (now holding 1 & 2) hands to 4.
    - Node 3 hands to 4.
    - Node 4 (now holding 1, 2, 3, & 4) hands to 8.

### 3. The Condition (parent <= n)
We stop at $N$ because there are no more "bosses" in our array. Any sum pushed beyond $N$ is "off the map" and doesn't matter for our queries.

### 4. Efficiency Secret
In $O(N \log N)$ build, each element travels to the top separately ($N \times \log N$ steps).
In $O(N)$ build, each element only moves **one level up** per loop iteration. Total steps = $N$.

#### Cpp implementation

```cpp
#include <vector>
#include <iostream>

using namespace std;

class FenwickTree {
private:
    vector<int> fenwick;
    int n;

    int rMsB(int x) {
        return x & -x;
    }

public:
    // O(N) Linear Build
    FenwickTree(const vector<int>& arr) {
        n = arr.size();
        fenwick.assign(n + 1, 0);

        // 1. Copy the values over (1-indexed)
        for (int i = 1; i <= n; i++) {
            fenwick[i] = arr[i - 1];
        }

        // 2. The Ripple Up: Each node adds its total to its immediate parent
        for (int i = 1; i <= n; i++) {
            int parent = i + rMsB(i);
            if (parent <= n) {
                fenwick[parent] += fenwick[i];
            }
        }
    }

    int sum(int pos) {
        int res = 0;
        while (pos > 0) {
            res += fenwick[pos];
            pos -= rMsB(pos);
        }
        return res;
    }

    void update(int pos, int delta) {
        while (pos <= n) {
            fenwick[pos] += delta;
            pos += rMsB(pos);
        }
    }
};
```
# The O(N) Fenwick Build: The "Relay Race" Physics

Building a Fenwick Tree in $O(N)$ is a "Senior-level" optimization. Instead of every element climbing to the top individually, we use a **Ripple Effect**.

---

### 1. The "Water Flow" Analogy
Imagine a series of buckets arranged in a hierarchy.
* Each bucket $i$ starts with its own "drop" of water (the original value `arr[i-1]`).
* Every bucket has **exactly one** immediate supervisor (parent).
* The parent of bucket $i$ is defined by the formula: `parent = i + (i & -i)`.

**The Physics:**
If you give your water to your boss, and then your boss gives all their collected water to *their* boss, eventually the CEO at the very top will hold the water from **everyone** in the chain below them.

---

### 2. Why "Immediate Boss" is Enough?
You might ask: *"Why don't I have to tell every boss above me manually?"* The secret lies in the **direction** of the loop. Because we process from $i = 1$ to $N$:

1.  **When $i=1$:** It gives its value to $i=2$.
2.  **When the loop reaches $i=2$:** Bucket 2 **already contains** the value from 1. 
3.  **Now $i=2$ acts:** It gives its **entire total** (1 + 2) to its boss, $i=4$.
4.  **When the loop reaches $i=4$:** It has already received the "bags" from 2 (which had 1) and 3. It then gives that combined "big bag" to $i=8$.

**By the time you reach the "Last Parent" ($< N$), that parent has accumulated the sums of every single subordinate below it.**

---

### 3. What happens if we stop early?
If you only passed the value to the first boss and didn't allow the "ripple" to continue, look at what happens to a **Query**:

* **The Scenario:** You call `query(8)`.
* **The Process:** The query loop looks directly at `bit[8]`.
* **The Failure:** If `bit[8]` didn't receive the "ripple" from `bit[4]`, and `bit[4]` didn't receive from `bit[2]`, then `bit[8]` would only contain its own single value.
* **Result:** Your sum would be wrong. The "Big Boss" nodes **must** be "fed" by the managers below them to maintain the $O(\log N)$ query speed.

---

### 4. Comparison Summary

| Method | Logic | Total "Steps" |
| :--- | :--- | :--- |
| **Standard (O(N log N))** | Each element climbs the whole tree alone. | $N \times \log N$ |
| **Linear (O(N))** | Each element hands a "baton" to the next person. | $N$ |

**The Mantra:** In $O(N)$ build, we don't move every brick to the roof; we move the floors into each other as we build upward.

class NumArray {
    vector<int>segtree;
    int n=0;
    int sz=0;

 void buildTree(vector<int>& nums,int s,int e,int i){

    if(s==e){
        segtree[i]=nums[s];
        return;
    }

    int mid=(s+e)/2;
    buildTree(nums,s,mid,2*i+1);
    buildTree(nums,mid+1,e,2*i+2);

    segtree[i]=segtree[2*i+1]+segtree[2*i+2];

 }

 void updateTree(int idx,int val,int s,int e,int i){

    if(s==e){
        segtree[i]=val;
        return;
    }
    int mid=(s+e)/2;
    if(idx<=mid) updateTree(idx,val,s,mid,2*i+1);
    else updateTree(idx,val,mid+1,e,2*i+2);

    segtree[i]=segtree[2*i+1]+segtree[2*i+2];
 }

int getSum(int l,int r,int s,int e,int i){

    if(r<s || e<l) return 0;

    if(l<=s && e<=r) return segtree[i];

    int mid=(s+e)/2;

    return getSum(l,r,s,mid,2*i+1)+getSum(l,r,mid+1,e,2*i+2);
}

public:
    NumArray(vector<int>& nums) {
        n=nums.size();
        sz=4*n;
        segtree.resize(sz);
        buildTree(nums,0,n-1,0);
    }
    
    void update(int index, int val) {

        updateTree( index, val,0,n-1,0);
        
    }
    
    int sumRange(int left, int right) {
        return getSum(left,right,0,n-1,0);
    }
};

### xor range fenwick tree

### Fenwick Tree for Range XOR (C++)

In C++, we use the `^=` operator for XOR updates and `i & -i` to isolate the least significant bit.

#### Implementation

```cpp
#include <iostream>
#include <vector>

class FenwickTreeXOR {
private:
    std::vector<int> tree;
    int n;

public:
    FenwickTreeXOR(int size) : n(size) {
        tree.assign(n + 1, 0);
    }

    // Standard Point Update: XORs the tree at index i with val
    // To replace a value at index i, pass (old_value ^ new_value)
    void update(int i, int val) {
        for (; i <= n; i += i & -i) {
            tree[i] ^= val;
        }
    }

    // Returns prefix XOR sum from 1 to i
    int query(int i) {
        int res = 0;
        for (; i > 0; i -= i & -i) {
            res ^= tree[i];
        }
        return res;
    }

    // Returns XOR sum of range [l, r] (1-based)
    int queryRange(int l, int r) {
        return query(r) ^ query(l - 1);
    }
};

int main() {
    std::vector<int> arr = {3, 5, 1, 2, 7};
    int n = arr.size();
    FenwickTreeXOR ft(n);

    // Initial construction
    for (int i = 0; i < n; i++) {
        ft.update(i + 1, arr[i]);
    }

    // Example Query: XOR of arr[0] to arr[2] (indices 1 to 3)
    // 3 ^ 5 ^ 1 = 7
    std::cout << "XOR range [1, 3]: " << ft.queryRange(1, 3) << std::endl;

    // Point Update: Change arr[1] (which is 5) to 10
    // We XOR the existing value with the new value: (5 ^ 10)
    int oldVal = 5, newVal = 10;
    ft.update(2, oldVal ^ newVal);

    // New Query: 3 ^ 10 ^ 1 = 8
    std::cout << "New XOR range [1, 3]: " << ft.queryRange(1, 3) << std::endl;

    return 0;
}
```
