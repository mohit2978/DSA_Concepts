### Binary Search STL Functions (C++ STL)

C++ provides three primary functions in the `<algorithm>` header to perform binary search on **sorted** containers.

---

### 1. `lower_bound`
Finds the first element in a range that is **not less than** (i.e., $\ge$) a given value.
* **Returns:** An iterator to the first element $\ge$ value.
* **If not found:** Returns an iterator to `end()`.

### 2. `upper_bound`
Finds the first element in a range that is **strictly greater than** (i.e., $>$) a given value.
* **Returns:** An iterator to the first element $>$ value.
* **If not found:** Returns an iterator to `end()`.



---

### 3. `std::binary_search`
A simple check to see if an element exists in a sorted range.
* **Returns:** `true` if the element exists, `false` otherwise.
* **Note:** It does not provide the position/index of the element.

---

### Comparison Example
Given a sorted vector: `v = {10, 20, 30, 30, 30, 40, 50}`  
Target Value: **30**

| Function | Logic | Iterator Points To | Index |
| :--- | :--- | :--- | :--- |
| **`lower_bound`** | First element $\ge 30$ | First `30` | `2` |
| **`upper_bound`** | First element $> 30$ | `40` | `5` |
| **`binary_search`** | Does `30` exist? | — | `true` |

Yes, you are absolutely correct. If you use `while(lo <= hi)` with assignments like `hi = mid` or `lo = mid`, you will hit an **Infinite Loop** when `lo == hi`.

Here is the exact mechanics of why this happens (The **"Black Hole"** Scenario):

---

### The Scenario: `lo` meets `hi`
Suppose your loop runs until `lo` and `hi` both point to the same index, say **5**.
* `lo = 5`
* `hi = 5`

Since you used `while(lo <= hi)`, the condition $5 \le 5$ is **True**, so we enter the loop one more time.

#### 1. The Calculation
We calculate `mid`:
$$mid = \frac{5 + 5}{2} = 5$$
So now, `lo`, `hi`, and `mid` are all **5**.

#### 2. The Trap
Now look at your update logic inside the loop:

* **Case A: `hi = mid`**
    * Code executes: `hi = 5`.
    * Result: `lo` is 5, `hi` is 5.
    * Next Iteration: $5 \le 5$ is **True**. `mid` is 5. `hi` becomes 5...
    * **Outcome:** Infinite Loop. The range never shrinks.
* **Case B: `lo = mid`**
    * Code executes: `lo = 5`.
    * Result: `lo` is 5, `hi` is 5.
    * Next Iteration: $5 \le 5$ is **True**. `mid` is 5. `lo` becomes 5...
    * **Outcome:** Infinite Loop.

---

### The Golden Rule of Binary Search Templates
The loop condition and the update logic are **coupled**. You must pick one of these three standard templates:

| Loop Condition | Update Logic | Why it works |
| :--- | :--- | :--- |
| `while(lo <= hi)` | `lo = mid + 1`<br>`hi = mid - 1` | Both updates exclude `mid`. Even if `lo == hi`, they cross each other (`lo` becomes `hi+1`), breaking the loop. |
| `while(lo < hi)` | `hi = mid`<br>`lo = mid + 1` | When `lo == hi`, the loop stops immediately. We never enter the "Black Hole" state where `hi = mid` does nothing. |
| `while(lo < hi)` | `lo = mid`<br>`hi = mid - 1` | **Warning:** This template requires `mid = (lo + hi + 1) / 2` (Ceiling), or it hangs when `lo` and `hi` are neighbors. |

---

### Summary
* **If you keep `mid` in the search space** (e.g., `hi = mid`), you **cannot** allow the loop to run when `lo == hi`. You must use `while(lo < hi)`.
* **If you discard `mid` from both sides** (`mid+1`, `mid-1`), you **must** use `while(lo <= hi)` to ensure you check that single last element.

Here is the clear breakdown of the **3 Standard Binary Search Templates**. The difference between them is purely about how you shrink the range and avoid infinite loops.

---

### Template 1: The "Exact Match" (Most Common)
Use this when you want to find a specific value. If you find it, you return immediately. If the loop finishes, the value doesn't exist.

* **Loop:** `while (lo <= hi)`
* **Updates:** Since we already checked `mid`, we discard it from both sides.
    * `lo = mid + 1`
    * `hi = mid - 1`
* **Termination:** Stops when `lo > hi` (Search space is empty).

---

### Template 2: The "First Valid" (Left-Biased)
Use this when you are looking for the **First** element that satisfies a condition (e.g., "First Bad Version", "Lower Bound", "Min in Rotated Array").

* **Loop:** `while (lo < hi)`
* **Updates:**
    * If `mid` satisfies condition: It might be the answer, but there could be an earlier one. We keep `mid` in the search space: `hi = mid`.
    * If `mid` fails: It is definitely not the answer. We discard it: `lo = mid + 1`.
* **Infinite Loop Safety:** Standard integer division `(lo + hi) / 2` rounds down. If `lo` and `hi` are neighbors (e.g., 4 and 5), `mid` becomes 4.
    * `lo = mid + 1` becomes 5 (Range shrinks).
    * `hi = mid` becomes 4 (Range shrinks).
    * **Safe.**
* **Termination:** Stops when `lo == hi`. The answer is at `lo`.

```cpp
// Scenario: Find the first bad version [G, G, G, B, B]
while(lo < hi) {
    int mid = lo + (hi - lo) / 2;
    if(isBadVersion(mid)) {
        hi = mid; // This is bad, but let's check left for earlier ones
    } else {
        lo = mid + 1; // This is good, throw it away
    }
}
return lo; // Points to the first Bad
```
---

### Template 3: The "Last Valid" (Right-Biased)
Use this when you want the **Last** element that satisfies a condition, or you need to keep `lo = mid`. This is the tricky one that causes infinite loops if you aren't careful.

* **Loop:** `while (lo < hi)`
* **Updates:**
    * If `mid` satisfies: We keep it, and try to go higher: `lo = mid`.
    * If `mid` fails: It's too high, discard it: `hi = mid - 1`.
* **Infinite Loop Safety:**
    * If `lo=4`, `hi=5` and we use standard division, `mid=4`. If the condition is true $\to$ `lo = mid` $\to$ `lo` stays 4. **Infinite Loop.**
    * **Fix:** We must round **UP** in the `mid` calculation: `(lo + hi + 1) / 2`.
    * Now `mid=5`. If true $\to$ `lo=5` (Range shrinks). If false $\to$ `hi=4` (Range shrinks).


```cpp

// Scenario: Find the last number < 5
while(lo < hi) {
    int mid = lo + (hi - lo + 1) / 2; // CEILING DIVISION!
    if(nums[mid] < 5) {
        lo = mid; // This is valid, but is there one to the right?
    } else {
        hi = mid - 1; // Too big
    }
}
return lo;
```

| Template | Condition | Updates | Mid Formula | Best Use Case |
| :--- | :--- | :--- | :--- | :--- |
| **Classic** | `lo <= hi` | `lo = mid + 1`<br>`hi = mid - 1` | `(L + H) / 2` | Exact Match / Simple Search |
| **Left Bias** | `lo < hi` | `lo = mid + 1`<br>`hi = mid` | `(L + H) / 2` | First True / Min in Rotated |
| **Right Bias** | `lo < hi` | `lo = mid`<br>`hi = mid - 1` | `(L + H + 1) / 2` | Last True / Max element |

## Death trap 

The place where your thinking likely goes wrong is assuming that **"Checking the condition"** (`lo <= hi`) and **"Moving the pointers"** (`hi = mid`) are two separate things. You think you can mix and match them.

**The Reality:** They are mechanically locked gears. If you change one, you break the other.

---

### The "Death Spiral" Trace
**Scenario:** Finding the Minimum in `[2, 1]`.
* **Indices:** 0, 1
* **Values:** 2, 1

You want to use `while(lo <= hi)` because it feels safer to check everything, and you want to use `hi = mid` because `mid` might be the minimum.

1.  **Iteration 1:**
    * `lo = 0, hi = 1`.
    * `mid = 0` (Value: 2).
    * **Is `arr[mid] > arr[hi]`?** Yes ($2 > 1$).
    * **Logic:** Minimum is to the right. `lo = mid + 1` $\to$ `lo = 1`.
    * *So far, so good.*

2.  **Iteration 2:**
    * `lo = 1, hi = 1`. (**Crucial Step:** `lo <= hi` is TRUE, so we enter).
    * `mid = 1` (Value: 1).
    * **Is `arr[mid] > arr[hi]`?** No ($1 > 1$ is False).
    * **Logic:** `mid` might be the minimum. We cannot discard it.
    * **Your Code:** `hi = mid`.
    * **Execution:** `hi` stays **1**.

3.  **Iteration 3:**
    * `lo = 1, hi = 1`. (TRUE).
    * `mid = 1`.
    * **Logic:** `hi = mid`. `hi` stays **1**.
    * **Result:** **Infinite Loop.** You are trapped here forever.

---

### What is wrong with the thinking?
The flaw in the mental model is thinking: *"The loop stops when I find the answer."*

* **Your Expectation:** "I found the min at index 1. The loop should stop."
* **The Reality:** The loop `while(lo <= hi)` **ONLY** stops when the pointers cross (`lo > hi`). It does not care if you found the answer.

**The Conflict:**
* `hi = mid` says: "I want to keep this index valid."
* `lo <= hi` says: "I will not stop until you discard all indices."

You are telling the computer: "Don't stop until the search space is empty," but simultaneously saying "Never empty the search space completely."

---

### How to fix the "Mental Model"
You have two choices to resolve this conflict. You must pick one; you cannot have both.

| Choice | Philosophy | Mechanism | Requirement |
| :--- | :--- | :--- | :--- |
| **A: Aggressive** | "I will discard everything, even the answer." | `lo <= hi` with `hi = mid - 1` | Must save `ans` in a variable before discarding. |
| **B: Conservative** | "I will never discard a potential answer." | `lo < hi` with `hi = mid` | Must stop before collision to avoid the "Death Spiral." |

---

### Summary
Your "wrong" thinking was trying to be **Conservative** (`hi = mid`) inside an **Aggressive** loop (`lo <= hi`).

* **Conservative Logic + Aggressive Loop** = Deadlock (Infinite Loop).
* **Aggressive Logic + Aggressive Loop** = Correct (but needs `ans` variable).

### 1. The "Aggressive" Approach (Harvest & Discard)
* **Philosophy:** "Discard `mid` immediately to shrink the range fast. Save the value before deleting it."
* **Loop:** `lo <= hi` (Runs until the range is empty).
* **Logic:** `hi = mid - 1` (Always throws away `mid`).
```cpp
class Solution {
public:
    int findMin(vector<int> &arr) {
        int lo = 0, hi = arr.size() - 1;
        int ans = INT_MAX; // Requirement: External storage

        // AGGRESSIVE LOOP: Runs until lo > hi
        while (lo <= hi) {
            int mid = (lo + hi) / 2;

            // Check if left side is sorted
            if (arr[lo] <= arr[mid]) {
                ans = min(ans, arr[lo]); // HARVEST: Save result
                lo = mid + 1;            // DISCARD: Throw away left half
            } else {
                ans = min(ans, arr[mid]); // HARVEST: Save result
                hi = mid - 1;             // DISCARD: Throw away mid & right
            }
        }
        return ans;
    }
};
```

**Key Requirement:** External storage (an `ans` variable) is required because the pointers will eventually move past the actual answer.

---

### 2. The "Conservative" Approach (Converge & Keep)
* **Philosophy:** "If `mid` might be the answer, keep it in the search space (`hi = mid`)."
* **Loop:** `lo < hi` (Stops when exactly 1 element remains).
* **Logic:** `hi = mid` (Keeps `mid` alive).

```cpp
class Solution {
public:
    int findMin(vector<int> &arr) {
        int lo = 0, hi = arr.size() - 1;

        // CONSERVATIVE LOOP: Stops when lo == hi
        while (lo < hi) {
            int mid = (lo + hi) / 2;

            // Check if right side is sorted
            // If mid > hi, the cliff is to the right
            if (arr[mid] > arr[hi]) {
                lo = mid + 1; // Discard mid (it's big, so it's not min)
            } 
            // If mid <= hi, the right side is smooth
            // So mid COULD be the min (or something to its left)
            else {
                hi = mid;     // KEEP: Do not subtract 1. Keep mid alive.
            }
        }
        // When loop ends, lo == hi, pointing to the answer
        return arr[lo];
    }
};
```

**Key Result:** No external variable is needed. When the loop ends, `lo` and `hi` point to the same index—the answer.

3.  **Update:** You do `hi = mid` (so `hi` becomes **5**).
4.  **Next Iteration:** `5 <= 5` is still **True**. 
5.  **Result:** The code enters the loop... **FOREVER**.

### The Correct Pairings (Memorize This)
To stay safe, you must always pair the **Loop Condition** with the **Update Logic** correctly:

| If your updates are... | You MUST use this loop... | Why? |
| :--- | :--- | :--- |
| **Aggressive** (`mid + 1`, `mid - 1`) | `while (lo <= hi)` | The `+1` / `-1` ensures indices **cross each other**, breaking the loop. |
| **Conservative** (`mid`, `mid + 1`) | `while (lo < hi)` | The strict `<` prevents the loop from running when `lo == hi`, avoiding the trap. |
