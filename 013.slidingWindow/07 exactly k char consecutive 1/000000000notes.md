```java
class Solution {
  private int solveAtmost(int[] arr, int tar) {
    int sum = 0;
    int n = arr.length;
    int si = 0, ei = 0;
    int ans = 0;
    while (ei < n) {
      sum += arr[ei++];
      while (sum > tar) {
        sum -= arr[si++];
      }
      ans += (ei - si);
    }
    return ans;
  }

  public int numSubarraysWithSum(int[] nums, int goal) {
    return solveAtmost(nums, goal) - (goal == 0 ? 0 : solveAtmost(nums, goal - 1));
  }
}

```
### The "Senior Engineer" Trap: The Limit of the Sliding Window

You have completely mastered the **Sliding Window** for subarray sums. But in an interview, the moment you write this flawless $O(N)$ code, the interviewer will lean forward and ask you one devastating question:

> **"What if the array has negative numbers?"** > *(e.g., `nums = [1, -1, 1, -1]`, `goal = 0`)*

The Sliding Window algorithm is entirely built on a single mathematical assumption: **Monotonicity.**

---

### 1. The Broken Assumption
In a window of positive integers:
* **Expanding (Right Pointer $j$++):** The sum is guaranteed to **grow** (or stay the same).
* **Shrinking (Left Pointer $i$++):** The sum is guaranteed to **shrink** (or stay the same).

This predictable behavior allows the two pointers to "chase" each other toward the solution. 

### 2. Why Negative Numbers Kill the Window
If you have negative numbers, expanding the window might suddenly cause your sum to **drop**. 
* If your `currentSum` is 10 and your `goal` is 15, you would normally expand. 
* But if the next number is **-5**, your sum drops to 5. You are now further away from your goal.
* The pointers lose their "sense of direction." They no longer know whether to expand to find a larger sum or shrink to find a smaller one. **The greedy nature of the Sliding Window completely breaks.**

---

### 3. The Pivot to Prefix Sums + HashMaps
When an interviewer introduces negative numbers, they are testing if you know when to **abandon** the Sliding Window. 

To solve "Subarray Sum Equals K" with negative numbers, you must switch to the **Prefix Sum + HashMap** architecture. 
* **The Math:** If $PrefixSum[j] - PrefixSum[i] = K$, then the subarray between $i$ and $j$ sums to $K$.
* **The Logic:** You store every $PrefixSum$ you’ve seen in a HashMap. At every step $j$, you check if $(PrefixSum[j] - K)$ already exists in your map.

---

### The Verdict

* **Sliding Window:** Best for **Positive Integers** (Non-negative). $O(N)$ Time, $O(1)$ Space.
* **Prefix Sum + HashMap:** Required for **Negative Integers**. $O(N)$ Time, $O(N)$ Space.

### The Prefix Sum + HashMap Pattern: The "Time Machine"

This is the **Prefix Sum + HashMap** pattern. It is the absolute silver bullet for any subarray problem involving negative numbers, and it is the exact solution for **LeetCode 560: Subarray Sum Equals K**.

When you introduce negative numbers, the Sliding Window loses its **"monotonicity."** Expanding the right pointer might actually make the sum smaller (by adding -5), and shrinking the left pointer might make the sum bigger (by dropping a -10). The pointers go blind.

To fix this, we abandon the pointers entirely and use a HashMap to build a **"Time Machine."**

---

### The Math: The "Chop Off" Strategy

Imagine you are walking through an array, keeping a running tally of every number you've seen so far. This is your `CurrentSum`. At any given moment, you want to know: *"Does a subarray ending right here equal my Target?"*

The math is beautifully simple:
$$CurrentSum - Target = PrefixSumToChopOff$$

If your `CurrentSum` is 14 and your `Target` is 8, you just need to know: *"Have I ever seen a running sum of 6 in the past?"* If you saw a 6 earlier, it means the numbers between that old 6 and your current 14 **must** add up to exactly 8.

---

### Why the HashMap is the "Time Machine"

Instead of moving pointers, we record every `PrefixSum` we encounter in a **HashMap**. 

1.  **Key:** The `PrefixSum` value.
2.  **Value:** How many times we have seen that specific sum.
3.  **The Check:** For every new number we add to our `CurrentSum`, we look back in our "memory" (the HashMap) to see if `CurrentSum - Target` has occurred before. 

---

### The Fatal "Zero" Mistake
In an interview, 90% of candidates forget the most important line of code in this pattern:
`map.put(0, 1);`

**Why?** If your `CurrentSum` hits exactly 8, and your `Target` is 8, then $8 - 8 = 0$. If you haven't initialized your map with a `0`, the algorithm won't "see" that the subarray starting from the very beginning of the array is a valid solution. You must always tell the map that you have seen a sum of `0` exactly once before you even start.

---

### The Final Verdict

* **Sliding Window:** Best for **Positive Integers** ($O(N)$ Time, $O(1)$ Space).
* **Prefix Sum + HashMap:** Required for **Negative Integers** ($O(N)$ Time, $O(N)$ Space).

**Would you like to see the implementation of this "Time Machine" logic for Subarray Sum Equals K?**


```java
import java.util.HashMap;
import java.util.Map;

class Solution {
    public int subarraySum(int[] nums, int k) {
        // HashMap to store: <PrefixSum, Frequency>
        Map<Integer, Integer> map = new HashMap<>();
        
        // THE TRAP: We must initialize the map with (0, 1).
        // This represents the "empty prefix" before we started iterating.
        map.put(0, 1);
        
        int currentSum = 0;
        int count = 0;
        
        for (int num : nums) {
            currentSum += num;
            
            // Do we have a previous sum we can chop off to get exactly 'k'?
            int neededPrefix = currentSum - k;
            if (map.containsKey(neededPrefix)) {
                count += map.get(neededPrefix); // Add how many times we saw it
            }
            
            // Record our current running sum into the history book
            map.put(currentSum, map.getOrDefault(currentSum, 0) + 1);
        }
        
        return count;
    }
}
```
### The Architectural Breakdown: Prefix Sums + HashMaps

This is the architectural breakdown of the **Prefix Sum + HashMap** pattern. It is the only way to solve subarray sum problems when negative numbers break the "monotonicity" of a standard Sliding Window.

---

### 1. The Zero Trap (`map.put(0, 1)`)
This is where **95% of candidates fail**. If your very first element is `3` and your target `k` is `3`, then `currentSum - k` equals `0`. 

* **The logic:** If `0` isn't in your map to represent the "starting line" (the empty subarray before index 0), you will completely miss valid subarrays that start from the very beginning. 
* **The fix:** Always initialize your map with `map.put(0, 1)` to account for the case where the `currentSum` exactly matches the target.

### 2. The Frequency Count
Why do we map `<PrefixSum, Frequency>` instead of just `<PrefixSum, Index>`? 

* In a problem like **Longest Subarray**, you only care about the first time you saw a sum (to maximize length). 
* In **Subarray Sum Equals K**, with negative numbers (e.g., `[3, -3, 3]`), your running sum might hit the exact same number multiple times. You need to know **how many** different starting points exist that could create your target sum, not just the most recent one.

---

### 3. Time & Space Complexity

| Metric | Complexity | Reason |
| :--- | :--- | :--- |
| **Time** | **$O(N)$** | You perform a single pass through the array, and HashMap `get/put` operations are $O(1)$ on average. |
| **Space** | **$O(N)$** | In the worst case (all unique prefix sums), the HashMap will store $N$ entries to keep the "history" of the walk. |

---

### The Final Verdict

You trade the **$O(1)$ space** of the Sliding Window for **$O(N)$ space**, but you gain the ability to instantly "jump back in time" and bypass the chaos of negative numbers.

You now have both subarray engines in your toolkit:
1.  **Sliding Window:** For purely positive/non-negative arrays ($O(1)$ space).
2.  **Prefix HashMap:** For arrays with negative numbers ($O(N)$ space).

class Solution {
public:
    int longestOnes(vector<int>& nums, int k) {
        int n=nums.size();
        int si=0,ei=0,count=0,len=0;
        while(ei<n){
            if(nums[ei++]==0) count++;
            if(count>k){
                if(nums[si]==0) count--;
                si++;
            }
            len=max(len,ei-si);
        }
        return len;
    }
};
